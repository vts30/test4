Key observations:

1. Memory Allocations (3.84 GB allocated total)

There is a massive allocation spike (~1.5 GB at once) at startup, then allocations drop significantly and stabilize
This is typical of a Spring Boot app initializing — loading beans, configurations, caches, reflection metadata
2. Survived Memory (2.3 MB survived)

Very little memory survives GC — meaning most objects are short-lived, which is good
But the survival spike that appears coincides with a GC event, suggesting a burst of long-lived objects was created at that moment
3. GC Survivor Space spike (image 1)

One clear GC collection event visible — Survivor Space filled briefly then emptied
Eden space pattern is flat → GC is keeping up with short-lived allocations most of the time
4. Metaspace / CodeHeap (images 2 & 3)

All stable and flat — no memory leak in non-heap areas
Metaspace ~150 MB — normal for a Spring Boot app of this size
Connecting to yesterday's CPU/I/O spike:

The large initial allocation burst + the 18.7 GB heap (from yesterday) together suggest:

The app allocates heavily at startup, filling Old Generation
Over time, Old Gen fills up → triggers a Full GC → causes the CPU 100% + I/O spike you saw at 14:00
Recommendation: Check Old Generation usage specifically — if it's consistently above 70–80%, Full GC is inevitable. Consider tuning -XX:NewRatio or switching to G1GC/ZGC which handle large heaps better.
